# 调度业务测试报告

日期：2026-04-11

本文档记录的不是 JUnit 方法级测试，而是五个真实进程启动后的业务链路验证结果。

## 一、测试目标

本次回答六个业务问题：

1. 单机模式能不能真实启动并持续执行
2. 单机模式修改 cron 和默认参数后能不能不重启直接生效
3. 中心模式 `STATIC` 下调度中心能不能真实下发到执行器客户端
4. 中心模式 `STATIC` 下任务更新和手工触发能不能真实生效
5. 中心模式 `DISCOVERY` 下客户端能不能通过 `spring-support-discovery-starter` 自动注册
6. 中心模式 `DISCOVERY` 下任务创建、统一下发、手工触发、热更新能不能真实生效

## 三、业务场景一：单机模式真实运行

### 场景说明

只启动 `spring-support-job-starter`，由当前服务自己轮询任务表并执行本地 `@Job("demoJob")`。

### 初始状态

```text
GET /job-local/api/demo/job/status
=> executeCount=0
```

### 验证 1：创建并启动单机任务

请求：

```text
POST /job-local/api/demo/job/setup?param=cron-start&autoStart=true
```

实际返回关键字段：

```text
jobId=1
jobNo=JOB202604111823000745777
jobScheduleTime=0/3 * * * * ?
jobDispatchMode=LOCAL
jobExecuteParam=cron-start
jobTriggerStatus=1
```

等待约 5 秒后再次查询：

```text
GET /job-local/api/demo/job/status
=> executeCount=7
=> lastParam=cron-start
```

日志观察到：

```text
任务创建成功: jobId=1, jobName=demo-local-job
本地任务执行成功: demoJob, jobNo=JOB202604111823000745777
```

结论：

- 单机模式能真实启动
- 本地调度链路已经实际跑起来
- 默认参数 `cron-start` 已进入真实执行

### 验证 2：手工触发单机任务

请求：

```text
POST /job-local/api/demo/job/trigger?param=manual-param
```

实际返回关键字段：

```text
accepted=true
executeCount=14
lastParam=manual-param
```

随后查询：

```text
GET /job-local/api/demo/job/status
=> executeCount=15
=> lastParam=cron-start
```

说明：

- 手工触发已经立即命中本地任务
- 下一次 cron 触发后，默认参数又回到 `cron-start`

结论：

- 单机模式同时支持“定时执行”和“人工立即执行”

### 验证 3：单机模式热更新

请求：

```text
POST /job-local/api/demo/job/setup?cron=0/2%20*%20*%20*%20*%20?&param=cron-updated&autoStart=true
```

更新返回关键字段：

```text
jobScheduleTime=0/2 * * * * ?
jobExecuteParam=cron-updated
```

随后查询：

```text
GET /job-local/api/demo/job/status
=> executeCount=138
=> lastParam=cron-updated
=> job.jobScheduleTime=0/2 * * * * ?
=> job.jobExecuteParam=cron-updated
```

结论：

- 单机模式已经验证“运行中修改 cron 生效”
- 单机模式已经验证“运行中修改默认参数生效”
- 本次验证不需要重启服务

## 四、业务场景二：中心模式真实运行

### 场景说明

启动两个独立进程：

1. 调度中心 `spring-api-support-scheduler-starter`
2. 执行器客户端 `spring-support-job-starter`

本次调度中心命名空间 `payment` 使用 `STATIC` 执行器配置，目标地址为：

```text
http://127.0.0.1:18081/payment/api
```

### 验证 1：中心创建远程任务

请求：

```text
POST /scheduler/api/v1/job-platform/payment/job
```

请求体关键字段：

```json
{
  "jobName": "demo-remote-job",
  "jobScheduleType": "CRON",
  "jobScheduleTime": "0/5 * * * * ?",
  "jobGlueType": "BEAN",
  "jobDispatchMode": "REMOTE",
  "jobExecuteBean": "demoJob",
  "jobExecuteParam": "remote-cron",
```

## 五、业务场景三：中心模式 DISCOVERY 真实运行

### 场景说明

启动两个独立进程：

1. Discovery 调度中心 `spring-api-support-scheduler-starter`
2. Discovery 执行器客户端 `spring-support-job-starter`

本次不再手工写死执行器地址，而是让客户端通过 `spring-support-discovery-starter` 注册到：

```text
/job-executor/payment
```

本轮真实联调用的是：

```text
multicast://230.0.0.1:18888
```

### 验证 1：Discovery 注册真实生效

执行器客户端启动完成后，中心日志持续出现：

```text
/job-executor/payment获取到1服务
```

同时中心调度日志里的实际触发地址是：

```text
http://192.168.1.11:18082/payment/api
```

结论：

- 客户端没有再依赖静态 `executors` 地址列表
- 调度中心已经通过 `spring-support-discovery-starter` 发现到真实执行器节点

## 七、补充结论

### 角色边界

- `spring-support-job-starter`
  - 单机模式下自己轮询执行
  - 中心模式下只做执行器客户端
- `spring-api-support-scheduler-starter`
  - 才是调度中心

### 执行器来源实现方式

当前 `executor-mode` 不是写死的 `if/switch` 创建，而是通过 `ServiceProvider` 加载 `SchedulerExecutorResolver` SPI 实现。

当前内置：

- `STATIC`
- `DISCOVERY`

当前行为：

- 默认模式是 `DISCOVERY`
- 缺少某个 SPI 对应依赖时，中心仍可启动，只是对应命名空间暂时解析不到执行器

### 关于 DISCOVERY 的真实联调边界

本次真实运行已经验证：

- `DISCOVERY + spring-support-discovery-starter + multicast` 已真实打通
- 调度中心能够自动发现客户端并完成真实下发

本次没有声称已经完成：

- `DISCOVERY + nacos`
- `DISCOVERY + consul`
- `DISCOVERY + zookeeper`
