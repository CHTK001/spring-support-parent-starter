# 控制台面板重构分步任务清单

## 文档说明

本文档用于记录按顺序执行的具体步骤。

规则：

- 每一轮只允许一个主步骤为 `进行中`
- 执行前先更新状态
- 执行后同步更新结果和遗留问题

## 当前阶段

阶段：`第一阶段 - 删除旧实现并落地 JDBC 第一版`

## 分步任务

### Step 1

状态：`已完成`

目标：

- 盘点旧 `monitor data-management` 相关代码范围

输出：

- 删除目标文件清单
- 受影响路由/API/菜单清单

完成标准：

- 三端旧 data-management 入口都已被定位

结果：

1. 前端旧代码主要集中在 `vue-support-monitor-starter/src/views/data-management/**`、`src/api/data-management/**`、路由模块 `data-management.ts` 与 `remaining.ts`
2. API 端旧代码主要集中在 `spring-api-support-monitor-starter` 的 `controller/data`、`service/data`、`engine/*SystemData*`、`entity/mapper/resources` 的 `SystemData*`
3. `spring-support-parent-starter` 当前未发现独立旧 data-management 模块，需在删除后编译时继续确认是否有悬挂引用

### Step 2

状态：`已完成`

目标：

- 删除旧 data-management 代码

输出：

- `spring-api-support-monitor-starter` 旧数据管理接口/服务删除
- `vue-support-monitor-starter` 旧页面/路由/API 删除
- `spring-support-parent-starter` 旧直接依赖删除

完成标准：

- 删除后无残留引用

当前进度：

1. 前端旧 data-management 页面、API、console 路由已删除
2. API 端 `SystemData*` 主线已删除
3. `Spider` 模块已退回本地 JDBC-only
4. `PerformanceMonitorAutoConfiguration` 已去除旧 data service 依赖
5. 仍需继续确认：
   - `spring-api-support-monitor-starter` 其他包中是否还有悬挂类型引用
   - `spring-support-parent-starter` 是否存在间接旧引用
   - 前端构建阻塞项是否与删除直接相关

结论：

1. 前端与 API 端旧 data-management 主链路源码已删除
2. 当前可进入 `Step 3`
